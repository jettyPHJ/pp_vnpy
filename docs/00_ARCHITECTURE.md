# 🏛️ 极简量化交易系统全局架构 (System Architecture)

> **Status:** BASELINE CANDIDATE (V1.6 Tick Execution Replay Aligned)
> **Notice:** 本文档定义了系统的最高宪法与不变量。任何对此架构边界的修改，必须经过显式的架构评审 (Revision)。

---

## 📜 宪法序言：为何基于 vn.py 进行二次开发 (Preamble & Motivation)

### 1. 业务使命 (The Mission)

本系统旨在建立一套 **高鲁棒性的中低频期货 CTA 交易底座** 。在实盘环境中，系统误差的破坏力远超策略回撤。系统的核心追求是：在任何软硬件故障或极端行情下，仓位状态与 PnL 永远 **可解释、可追溯、可对账** 。

### 2. 原生框架的定位与二次开发边界 (Context & Limitations)

原生 `vn.py` 是一款极为优秀的通用量化框架，但在应对**机构级准实盘/实盘的连续交易**时，我们需要在以下五个方向进行底层重构：

* **逻辑与物理的解耦** ：将投研逻辑与执行物理边界严格剥离。
* **执行摩擦的"参数化"陷阱** ：执行摩擦必须作为独立的模块被挂载，消除"回测乐观主义"。
* **状态机与对账闭环** ：建立独立于订单流的客观对账机制与仓位自愈能力。
* **执行全链路溯源** ：建立 TraceID，打通发单指令到底层 API 之间的全局追踪。
* **风控的强制管道化** ：风控必须升级为发单的"必经关卡"。

## 🚨 系统长期不变量 (System Invariants)

1. **逻辑/物理分离原则** ：策略层表达意图，风控层执行裁决，执行层完成翻译。各层严禁越权代办。
2. **职责三角划分 (Price vs. Cost vs. Hub)** ：

* **执行层 (Execution)** ：只负责生产"物理事实"（计算并在价格域产生偏差，如 `execution_price`）。
* **对账层 (Ops/Accounting)** ：负责"财务核算"（将价格偏差转换为资金成本金额，如 `slippage_cost`）。
* **引擎层 (Engine)** ：仅作为流转与数据下发枢纽。引擎负责从数据层提取静态参数（如 `contract_multiplier`），并向下透传给执行层和对账层，**严禁引擎隐式承担金额核算，也严禁对账层自行向上查找静态数据。**

1. **单一发单管道** ：所有生产订单必须通过统一 Pipeline。
2. **仓位唯一真相来源** ：`actual_pos` 只能由真实的成交事件严格按照多空符号更新。
3. **Bar 决策 / Tick 执行分离原则** ：策略基于连续合约 Bar 生成逻辑意图；执行层基于物理合约 Tick 进行回放验证。**Tick 数据只允许在订单生成之后参与成交判断，绝对不允许反向参与当前信号形成、指标计算、状态识别与连续合约构建，严防未来函数。**

## 🏗️ 6 大核心逻辑层定义

### 1. 投研与仿真层 (Research Layer)

* **核心职责** ：历史数据回放仿真、参数优化、统计报告。V1.6 起显式支持 Bar（策略决策）与 Tick（执行回放）双轨时间线的并行驱动。

### 2. 策略与信号层 (Strategy Layer)

* **核心职责** ：承载 Alpha 逻辑，生成纯逻辑交易意图（`SignalOrder`）并赋予追溯码 `chain_id`。
* **边界约束** ：只负责写入目标逻辑持仓，严禁在生产策略中处理物理合约的映射转换；严禁读取 Tick 执行回放的成交细节来反向影响信号生成。

### 3. 风控层 (Risk Layer)

* **核心职责** ：发单的强制物理关卡。实施 **硬约束（Hard Constraints）** ，包括资金不足拒绝、单笔最大手数拦截；计算风险额度、执行系统级流控。只有通过/驳回/收缩（allow / reject / shrink）的权力，绝不修改成交价格。

### 4. 执行路由层 (Execution Layer)

* **核心职责** ：系统对接外界的统一物理执行入口与摩擦模拟中心。
* **边界约束** ：负责实施 **软约束（Soft Constraints）** 。将风控放行的逻辑订单，结合 **市场上下文（Market Context）** ，转化为包含**模型化滑点成本**的物理订单。V1.6 起新增 `TickExecutionModel`，在物理合约 Tick 路径上进行高保真成交回放，支持主动/被动成交识别与部分成交（Partial Fill）。

### 5. 状态与对账层 (Ops & Reconciliation Layer)

* **核心职责** ：维护三仓位模型隔离（`target_pos`, `allowed_pos`, `actual_pos`）；管理全生命周期交易审计日志；执行内存防爆回收；Broker 账单核对与容灾对账。V1.6 起须额外归档 Tick 执行质量指标（主动/被动成交比、部分成交次数、成交价相对 mid-price 偏移）。

### 6. 数据资产层 (Data Layer)

* **核心职责** ：提供连续合约映射表、换月日历、行情与特征库。只提供客观事实，不维护主观交易状态。严格切分"连续合约 Bar"与"物理合约 Tick"的获取和加载路径，确保两者互不污染。
