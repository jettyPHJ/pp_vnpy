# 🏛️ 极简量化交易系统全局架构 (System Architecture)

> **Status:** BASELINE CANDIDATE
> **Notice:** 本文档定义了系统的最高宪法与不变量。任何对此架构边界的修改，必须经过显式的架构评审 (Revision)。

---

## 📜 宪法序言：为何基于 vn.py 进行二次开发 (Preamble & Motivation)

### 1. 业务使命 (The Mission)

本系统旨在建立一套**高鲁棒性的中低频期货 CTA 交易底座**。在实盘环境中，系统误差的破坏力远超策略回撤。系统的核心追求是：在任何软硬件故障或极端行情下，仓位状态与 PnL 永远**可解释、可追溯、可对账**。

### 2. 原生框架的定位与二次开发边界 (Context & Limitations)

原生 `vn.py` 是一款极为优秀的通用量化框架，极大地降低了投研门槛。但在应对**机构级准实盘/实盘的连续交易**时，我们需要在以下五个方向进行底层重构：

* **逻辑与物理的解耦**：原生策略强绑定物理合约（如 `rb2410.SHFE`），处理期货自动换月、主连映射或跨券商适配时会引发逻辑侵入。必须将投研逻辑与执行物理边界严格剥离。
* **执行摩擦的“参数化”陷阱**：原生框架将滑点、手续费视为静态参数，无法模拟**执行摩擦随挂单深度、成交量动态变化**的冲击成本。这会导致严重的“回测乐观主义”。执行摩擦必须作为独立的模块被挂载。
* **状态机与对账闭环**：原生仓位维护依赖内存简单累加。为防范断线重连导致的“幽灵仓位”，必须建立独立于订单流的客观对账机制与仓位自愈能力。
* **执行全链路溯源**：发单指令到底层 API 之间缺乏全局追踪码（TraceID），导致复杂场景下（止损、换月、策略混合）交易动作无法归因。
* **风控的强制管道化**：风控不能是“旁路插件”，必须升级为发单的“必经关卡”。

## 🚨 系统长期不变量 (System Invariants)

1. **逻辑/物理分离原则 (Mapping Separation)**：**生产路径**中的策略不得依赖物理合约代码。所有逻辑合约到物理合约的转换，必须由执行路由层统一完成。
2. **执行摩擦模块化原则 (Friction Isolation)**：严禁在策略或引擎主逻辑中硬编码滑点和费率。所有摩擦模拟必须通过 `Execution Layer` 调用的独立摩擦模型（Friction Model）计算。
3. **单一发单管道 (Single Pipeline Constraint)**：所有生产订单（含手工、止损、换月）**必须**通过统一 Pipeline。严禁绕过 Pipeline 直接调用底层接口。
4. **仓位唯一真相来源 (Single Source of Truth)**：`actual_pos` 只能由真实的成交事件严格按照多空符号更新：
   * **回测环境**：由模拟撮合生成的 `TradeData` 增量更新。
   * **实盘环境**：由 Broker/Gateway 回报的 `TradeData` 增量更新。
   * **恢复环境**：允许由独立对账服务通过 Broker Snapshot 发起全量校准，但必须显式记录审计事件。
5. **职责单向流动 (Unidirectional Dependency)**：策略层表达意图，风控层执行裁决，执行层完成翻译。各层严禁越权代办。

## 🏗️ 6 大核心逻辑层定义

### 1. 投研与仿真层 (Research Layer)

* **核心职责**：历史数据回放仿真、参数优化、统计报告。

### 2. 策略与信号层 (Strategy Layer)

* **核心职责**：承载 Alpha 逻辑，生成纯逻辑交易意图（`SignalOrder`）并赋予追溯码 `chain_id`。
* **边界约束**：只负责写入目标逻辑持仓，严禁在生产策略中处理物理合约的映射转换。

### 3. 风控层 (Risk Layer)

* **核心职责**：发单强制关卡。决策信号是否允许通过 (allow / reject)、计算风险额度、执行系统级流控。

### 4. 执行路由层 (Execution Layer)

* **核心职责**：系统对接外界的**统一物理执行入口**、**翻译枢纽**与**摩擦模拟中心**。
* **边界约束**：负责物理合约映射转换、**执行摩擦模型挂载（滑点/费率）**、Tick Size 对齐。不包含业务风控。

### 5. 状态与对账层 (Ops & Reconciliation Layer)

* **核心职责**：维护三仓位模型隔离（`target_pos`, `allowed_pos`, `actual_pos`）；管理全生命周期交易审计日志；执行内存防爆回收；Broker 账单核对与容灾对账。

### 6. 数据资产层 (Data Layer)

* **核心职责**：提供连续合约映射表、换月日历、行情与特征库。只提供客观事实，不维护主观交易状态。
