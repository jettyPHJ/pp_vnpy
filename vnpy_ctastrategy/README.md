# vn.py 期货回测引重构 - 项目说明

---

## 1. 为什么要重构：原版引擎在期货 CTA 回测中的关键缺口

原版 `vnpy.trader.backtesting` 在单合约、理想化条件下表现良好，但在**国内商品期货 CTA（主力换月 + 夜盘）**场景下，存在若干结构性缺口，导致回测结果系统性偏乐观、难以与实盘对齐。

### 1.1 原版关键缺口

| 序号 | 核心缺口             | 重要性     | 典型后果                                             |
| ---- | -------------------- | ---------- | ---------------------------------------------------- |
| 1    | 物理真实账本缺失     | ★★★★★ | 换月未执行“平旧开新”，摩擦成本缺失，旧合约挂单残留 |
| 2    | 夜盘交易日对齐错位   | ★★★★★ | 日度盈亏、Sharpe、胜率统计失真                       |
| 3    | 成本与执行模型硬编码 | ★★★★   | 无法扩展动态滑点、延迟、部分成交等                   |
| 4    | 资金与保证金约束缺失 | ★★★     | 回测默认总能成交，无法模拟资金不足拒单               |
| 5    | TICK/BAR 边界不清晰  | ★★★     | 高频场景下物理连续逻辑容易混乱                       |

### 1.2 缺口 → V1 → V2 映射关系

| 核心缺口             | V1 已解决范围                               | V2 补完方向                        |
| -------------------- | ------------------------------------------- | ---------------------------------- |
| 物理真实账本缺失     | 建立物理仓位账本 + 信号/执行分离 + 换月摩擦 | 引入流动性约束、部分成交           |
| 夜盘交易日对齐错位   | 修正交易日归属                              | 无需结构升级，仅需实盘验证         |
| 成本与执行模型硬编码 | 抽象四大模型接口                            | 动态滑点、延迟、部分成交、阶梯费率 |
| 资金与保证金约束缺失 | 预留 MarginModel 接口                       | 动态权益、保证金占用、资金不足拒单 |
| TICK/BAR 边界不清晰  | 明确限制不支持场景                          | 构建 TICK 级物理连续处理           |

## 2. V1 的目标与已解决的问题

### 2.1 V1 的定位

V1 的目标不是“完全复现实盘”，而是：

> 在保持原有回测流程可用的前提下，解决最影响期货 CTA 回测可信度的结构性问题，并为后续校准提供清晰扩展接口。

### 2.2 V1 相对原版已解决的问题

| 问题                | V1 解决方式                                                           |
| ------------------- | --------------------------------------------------------------------- |
| 换月成本缺失        | 引入 ContinuousBuilder + 物理合约路由，换月时真实扣除滑点与双边手续费 |
| 夜盘统计错位        | 新增 `_get_trading_date()`，正确映射夜盘/凌晨交易日                 |
| 仓位与执行混用      | 引入物理仓位账本（`physical_positions`），信号层与执行层分离        |
| 挂单残留问题        | 换月时自动撤销旧主力上的限价单与止损单                                |
| 成本/执行逻辑耦合   | 抽象出 Execution / Slippage / Commission / Margin 四类模型接口        |
| TICK/BAR 边界不清晰 | TICK + 物理连续场景显式抛出 NotImplementedError                       |
| 成交边界保护不足    | 支持 PARTTRADED 状态与成交量边界保护                                  |

### 2.3 V1 当前能力边界

V1 已可作为：

- 期货 CTA 单策略研究底座
- 多品种基线策略构建基础
- 回测 vs 模拟盘对齐分析工具

但仍未覆盖：

- 部分成交的真实流动性建模
- 延迟模型
- 真实盘口与成交量约束
- 完整保证金占用与资金冻结机制
- TICK 模式下的物理连续合约回测

## 3. 如何验证 V1 确实解决了这些问题

**推荐测试配置**
品种：螺纹钢（rb）、白银（ag）、股指（IF）
数据区间：覆盖至少 2 次主力换月
测试策略：Donchian Channel（首选）、双均线（对照）

### 验证清单

| 类别           | 验证项          | 验证标准                                                          |
| -------------- | --------------- | ----------------------------------------------------------------- |
| 换月行为验证   | 主力切换与摩擦  | 触发切换日志、`rollover_pnl` 为负值、旧挂单被清理、净值平稳过渡 |
| 夜盘归属验证   | 交易日映射      | 周五夜盘归入下周一、凌晨数据归入正确交易日                        |
| 成本敏感性验证 | 滑点/手续费影响 | 放大成本后收益下降、回撤上升，但交易逻辑不变                      |
| 行为合理性验证 | 趋势 vs 震荡    | 趋势行情整体盈利、震荡行情出现磨损或回撤                          |
| 可复现性验证   | 多次回测        | 同一参数多次运行，结果必须完全一致                                |

## 4. V2 预留接口与后续演进方向

### V2 主要演进方向

| 模块              | V1 当前状态          | V2 计划实现                            |
| ----------------- | -------------------- | -------------------------------------- |
| ExecutionModel    | 基础撮合逻辑         | 延迟、部分成交、成交量限制、流动性建模 |
| SlippageModel     | 固定滑点             | 动态滑点（ATR、成交量、时段等）        |
| CommissionModel   | 固定费率             | 品种差异、开平差异、平今惩罚、阶梯费率 |
| MarginModel       | 预留接口（默认通过） | 动态权益、保证金占用、资金不足拒单     |
| ContinuousBuilder | BAR 模式完整支持     | TICK 级物理连续合约拼接                |
| Portfolio / Risk  | 暂未内建             | 组合级资金分配、风险预算、仓位约束     |

**V2 目标**：让回测结果与模拟盘/实盘的主要偏差更少、更可解释、可持续校准。
